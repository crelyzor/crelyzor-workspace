# Phase 4.6 — Infrastructure Optimization

**Date:** 2026-04-26
**Status:** Design approved, ready for implementation
**Prerequisite:** None — independent of billing/monetization work

---

## Problem

The current infrastructure wastes money and has a critical failure mode:

1. **Upstash Redis free tier exhausted in 3 days** — 502K/500K commands used. Bull creates 3 Redis connections per queue × 5 queues × 2 processes (API + worker) × 2 environments (staging + prod) = 60 persistent connections all polling 24/7 on one free-tier instance, even with zero jobs.

2. **Staging VM runs 24/7 for no reason** — 6 containers burning ~₹2,050/month on an e2-medium with near-zero traffic. The staging worker polls Redis continuously for jobs that never come.

3. **No Docker resource limits** — on 4GB RAM VMs, any container can OOM-kill everything. No memory caps, no CPU limits.

4. **Bloated Docker images** — full `node_modules` (including devDependencies) copied into production backend and public images. Next.js public site installs pnpm globally in prod image just to run `pnpm start`.

5. **Full rebuild on every deploy** — `deploy.sh` rebuilds all 5 images regardless of what changed. A frontend CSS fix triggers backend + worker + public rebuilds.

6. **Both environments share one Upstash Redis** — staging and prod Bull queues use the same `usable-buck-40108.upstash.io`. Queue names collide (e.g., both schedule `daily-digest-cron`). Combined idle polling doubled the command burn.

## Design

### 1. Local Redis container (replaces Upstash)

Add `redis:7-alpine` to both `docker-compose.prod.yml` and `docker-compose.staging.yml`.

```yaml
redis:
  image: redis:7-alpine
  restart: always
  command: redis-server --maxmemory 128mb --maxmemory-policy allkeys-lru --save ""
  volumes:
    - redis_data:/data
  networks:
    - internal
  healthcheck:
    test: ["CMD", "redis-cli", "ping"]
    interval: 10s
    timeout: 5s
    retries: 5
```

Configuration choices:
- **`--maxmemory 128mb`** — caps Redis at 128MB. More than enough for Bull queues + rate limiting. Protects the 4GB VM.
- **`--maxmemory-policy allkeys-lru`** — evicts oldest keys when full. Rate limit counters and Bull metadata are safe to evict.
- **`--save ""`** — disables RDB snapshots. Bull jobs are transient — if Redis restarts, pending jobs get re-created. No need for persistence.
- **Persistent volume** (`redis_data`) — survives `docker compose down` but not `docker compose down -v`. Keeps repeatable jobs (cron schedules) across restarts.

Environment variable changes:
```bash
# Before (in .env.prod / .env.staging)
REDIS_URL=rediss://default:***@usable-buck-40108.upstash.io:6379
UPSTASH_REDIS_REST_URL=https://usable-buck-40108.upstash.io
UPSTASH_REDIS_REST_TOKEN=***

# After
REDIS_URL=redis://redis:6379
# UPSTASH_REDIS_REST_URL — removed
# UPSTASH_REDIS_REST_TOKEN — removed
```

### 2. Replace Upstash REST client with ioredis

`src/config/redisClient.ts` currently uses `@upstash/redis` (REST-based). Replace with an ioredis singleton that reuses the same `REDIS_URL` as Bull.

**Before:**
```typescript
import { Redis } from "@upstash/redis";
export function getRedisClient(): Redis { ... }
```

**After:**
```typescript
import IORedis from "ioredis";

let _client: IORedis | null = null;

export function getRedisClient(): IORedis {
  if (_client) return _client;
  const url = process.env.REDIS_URL;
  if (!url) throw new Error("REDIS_URL is required");
  _client = new IORedis(url, { maxRetriesPerRequest: null, enableReadyCheck: false });
  return _client;
}
```

Callers (`authMiddleware.ts`, `googleCalendarService.ts`, `aiService.ts`) use `.incr()`, `.expire()`, `.ttl()`, `.set()`, `.get()` — all identical API between `@upstash/redis` and `ioredis`. No caller changes needed.

After this, remove `@upstash/redis` from `package.json`.

### 3. Consolidate 5 queues → 1 queue

All 5 queues use the same Redis, same retry patterns, and jobs are already differentiated by name. Consolidating to 1 queue reduces worker connections from 15 to 3.

**Before:**
```
QueueNames.TRANSCRIPTION       → 3 connections
QueueNames.AI_PROCESSING       → 3 connections
QueueNames.RECALL_BOT_DEPLOY   → 3 connections
QueueNames.RECALL_RECORDING    → 3 connections
QueueNames.EMAIL               → 3 connections
                                 ─────────────
                                 15 connections
```

**After:**
```
QueueNames.MAIN                → 3 connections
```

`queue.ts` changes:
- Single `QueueNames.MAIN = "crelyzor"` enum value
- Single queue instance with unified config
- Per-job settings (attempts, backoff, lockDuration) specified at `.add()` time, not queue level
- All getter functions (`getTranscriptionQueue`, `getAIProcessingQueue`, etc.) return the same queue instance — API stays the same for callers
- `JobNames` constants unchanged — jobs differentiated by name, not queue

`jobProcessor.ts` changes:
- All `.process()` calls register on the single queue with their job name
- No logic changes — just `queue.process(JobNames.TRANSCRIBE, ...)` instead of `transcriptionQueue.process(...)`

### 4. Producer-only mode for API server

The API server (`index.ts`) only adds jobs — it never processes them. It should create zero polling connections.

`initializeProducerQueues()` creates the single queue with a shared ioredis connection. No subscriber, no bclient, no polling.

**Connection count after:**
| Process | Connections |
|---------|-------------|
| API server | 1 (shared ioredis for .add() + rate limiting) |
| Worker | 3 (client + subscriber + bclient for 1 queue) |
| **Total per VM** | **4** |

### 5. Docker resource limits

Add memory and CPU limits to all containers in both prod and staging compose files:

```yaml
backend:
  deploy:
    resources:
      limits:
        memory: 512M
        cpus: "0.5"

worker:
  deploy:
    resources:
      limits:
        memory: 1G      # AI/transcription jobs can spike
        cpus: "0.5"

postgres:
  deploy:
    resources:
      limits:
        memory: 512M
        cpus: "0.5"

redis:
  deploy:
    resources:
      limits:
        memory: 128M
        cpus: "0.25"

frontend:
  deploy:
    resources:
      limits:
        memory: 64M
        cpus: "0.15"

public:
  deploy:
    resources:
      limits:
        memory: 256M
        cpus: "0.3"

nginx:
  deploy:
    resources:
      limits:
        memory: 64M
        cpus: "0.15"
```

Total memory budget: 512 + 1024 + 512 + 128 + 64 + 256 + 64 = **2,560MB** — leaves ~1.5GB for OS + buffer on 4GB VM.

### 6. Slim Docker images

**Backend Dockerfile:**
```dockerfile
# Add after builder stage, before runner
FROM deps AS pruned
RUN pnpm prune --prod

# Runner copies from pruned instead of builder
COPY --from=pruned /app/node_modules ./node_modules
```

**Public (Next.js) Dockerfile:**
Use Next.js standalone output — produces a self-contained `server.js` that doesn't need node_modules.

```javascript
// next.config.js (or .ts)
output: "standalone"
```

```dockerfile
FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
COPY --from=builder /app/public ./public
EXPOSE 5174
CMD ["node", "server.js"]
```

This removes the `RUN npm install -g pnpm` from the runner stage and drops image size significantly.

### 7. Smarter deploys

Update `deploy.sh` to detect which repos changed and only rebuild those services:

```bash
# Save current HEAD before pulling
PREV_BACKEND=$(git -C ./crelyzor-backend rev-parse HEAD 2>/dev/null)
PREV_FRONTEND=$(git -C ./crelyzor-frontend rev-parse HEAD 2>/dev/null)
PREV_PUBLIC=$(git -C ./crelyzor-public rev-parse HEAD 2>/dev/null)

# ... (git fetch + reset happens here) ...

# Check which repos changed by comparing saved HEAD to new HEAD
CHANGED_SERVICES=""
[ "$(git -C ./crelyzor-backend rev-parse HEAD)" != "$PREV_BACKEND" ] && CHANGED_SERVICES="$CHANGED_SERVICES backend worker"
[ "$(git -C ./crelyzor-frontend rev-parse HEAD)" != "$PREV_FRONTEND" ] && CHANGED_SERVICES="$CHANGED_SERVICES frontend"
[ "$(git -C ./crelyzor-public rev-parse HEAD)" != "$PREV_PUBLIC" ] && CHANGED_SERVICES="$CHANGED_SERVICES public"

# Only rebuild changed services (or all on first deploy / infra change)
if [ -n "$CHANGED_SERVICES" ]; then
  docker compose -f "$COMPOSE_FILE" build $CHANGED_SERVICES
else
  docker compose -f "$COMPOSE_FILE" build  # fallback: rebuild all
fi
docker compose -f "$COMPOSE_FILE" up -d
```

### 8. Staging optimization

Remove the worker service from `docker-compose.staging.yml`. Staging doesn't need 24/7 job processing. When testing queue jobs on staging, start the worker manually:

```bash
docker compose -f docker-compose.staging.yml run --rm worker
```

This saves one container's worth of memory + eliminates 3 Redis polling connections on staging.

## Rollback

If local Redis causes issues, revert to Upstash by:
1. Restoring `REDIS_URL` / `UPSTASH_*` env vars
2. Re-adding `@upstash/redis` to package.json
3. Reverting `redisClient.ts` to Upstash REST client
4. Removing redis service from Docker Compose

Queue consolidation and producer-only mode are independent improvements — they work with both local and external Redis.
